3. Giao thức giữa background/background.js và content/*.js
  Xét ở phía background.js: chrome.runtime.onMessage.addListener(((msg, sender, sendResponse)) => {})
  (msg là nhận, chi tiết xem phần handlers)
  msg = {
          type, // thuộc các dạng "LOG", "SOURCE.", "SUB." (có thể bổ sung sau)
          payload
        }
    3.1. type === "LOG"
            payload = { type, text, url, timestamp }
          sendResponse (tổng): { type: 'LOGGED' }

    3.2. type === "SOURCE.ADD"
            payload = { url }
          sendResponse (tổng): { type: 'SOURCE.ADDED', payload: folderData }

    3.3. type === "SOURCE.GET_ALL"
            payload = undefined
          sendResponse (tổng): { type: 'SOURCE.LIST', payload: getSourceList() }; // xem storage.js

    3.4. type === "SOURCE.REMOVE"
            payload = { savedAt }
          sendResponse (tổng): { type: 'SOURCE.REMOVED', payload: await removeSource(payload.savedAt)}; // xem storage.js

    3.5. type === "SUB.SEARCH"
            payload = { videoId }
          sendResponse (tổng): resolveSubtitles(videoId)
              { type: 'SUB.READY', payload: cached }
              { type: 'SUB.EMPTY', payload: [] }
              { type: 'SUB.WAIT', payload: candidates }

    3.4. type === "SUB.SELECT"
            payload = { videoId, candidate }
          sendResponse (tổng): processSubtitles(videoId, candidate, rawText)
              { type: 'SUB.READY', payload: subObj }

    3.5. type === "SUB.GET_ALL"
            payload = undefined
          sendResponse (tổng): { type: 'SUB.LIST', payload: await getSubDataList() }
    
    3.6. type === "SUB.REMOVE"
            payload = { videoId }
          sendResponse (tổng): { type: 'SUB.REMOVED', payload: await removeSubData(videoId) };

    3.7. type === "SUB.LOCAL"
            payload = { videoId, rawText, fileName }
          sendResponse (tổng): processSubtitles(videoId, localFileData (xử lí từ fileName), rawText)
    3.8. Dữ liệu khác chuẩn/chạy handlers bị lỗi
          sendResponse (tổng): { type: 'ERROR', payload: err.message hoặc 'Unknown action' }

🛠️ 2. Ma trận quyết định & Trạng thái phản hồi (Decision Matrix)
Khi content.js gửi yêu cầu tìm kiếm phụ đề qua `SUBTITLE.SEARCH`, background.js điều phối tìm kiếm và phản hồi 1 trong 3 trạng thái:
┌──────────────┬──────────────────────────────────────────────────────────────────────────────┐
│ Trạng thái   │ Ý nghĩa & Hành động tiếp theo                                                │
├──────────────┼──────────────────────────────────────────────────────────────────────────────┤
│ HIT          │ Có sẵn dữ liệu trong Cache hoặc tự động giải quyết nếu chỉ có 1 file tìm thấy│
│              │ -> Phản hồi lập tức gửi kèm dữ liệu cấu trúc (parsedData hoặc raw text).     │
├──────────────┼──────────────────────────────────────────────────────────────────────────────┤
│ MULTIPLE     │ Tìm thấy nhiều file phụ đề ứng với videoId hiện tại                          │

│ EMPTY        │ Không tìm thấy file phụ đề nào trên các nguồn cấu hình                       │

📡 3. Chi tiết Giao thức Giao tiếp RPC (Message Protocols v1.0)
Tất cả các file trao đổi dữ liệu xuyên suốt extension thông qua chuẩn format tin nhắn sau:
- Gói tin gửi đi: { id: string, type: string, payload: object }
- Gói tin phản hồi: { id: string, status: 'HIT' | 'MULTIPLE' | 'EMPTY' | 'ERROR', data?: object, candidates?: array, message?: string }

Danh sách các Type hành động chuẩn hóa:
1. 'SUBTITLE.SEARCH':
   - Payload: { videoId: "chuỗi_id_youtube" }
   - Hành động: background.js quét Cache trước, nếu hụt sẽ gọi fetcher.js quét GitHub/GDrive.
2. 'SUBTITLE.SELECT':
   - Payload: { videoId: "chuỗi_id_youtube", candidate: { id, fileName, url, sourceType } }
   - Hành động: Thực hiện tải file thô, tự động phân tích (nếu là BACKGROUND mode), ghi Cache và trả về dữ liệu HIT.
3. 'SOURCE.GET_ALL':
   - Payload: {}
   - Hành động: Lấy toàn bộ danh sách các link nguồn lưu trong Chrome Storage.
4. 'SOURCE.REMOVE':
   - Payload: { id: "uuid_nguồn" }
   - Hành động: Xóa nguồn khỏi cấu hình lưu trữ.
   
🎨 3. Yêu cầu đặc tả cho UI & Styling (Giao diện hiển thị)
Để đảm bảo extension không phá vỡ giao diện của YouTube (vốn thay đổi DOM liên tục) và mang lại trải nghiệm mượt mà, chúng ta tách biệt hoàn toàn CSS và logic UI.
3.1. styles.js (CSS Injector Engine)
Nhiệm vụ: Tiêm (inject) các rule CSS cần thiết vào YouTube DOM.
Quy tắc thiết kế (Guardrails):
No Global Bleeding: Tất cả class/id phải có prefix đặc trưng (ví dụ: #ass-cee-overlay-container, .ass-cee-btn, .ass-cee-chooser-modal) để không bị CSS của YouTube đè lên hoặc làm hỏng UI của YouTube.
Z-Index Hierarchy: - Layer phụ đề (#ass-cee-overlay-container) phải nằm trên thẻ <video> nhưng phải nằm dưới thanh điều khiển (.ytp-chrome-bottom) để người dùng vẫn bấm pause/play được.
Cần có thuộc tính pointer-events: none; cho Layer phụ đề để các click chuột xuyên qua rơi xuống đúng Video Player.
Chooser UI (Bảng chọn sub) phải có z-index cao nhất để dễ thao tác, có nền tối (Dark mode) đồng điệu với phong cách của YouTube.
3.2. ui.js (Chooser UI & Button Logic)
Nhiệm vụ: Hiển thị danh sách phụ đề khi background.js trả về trạng thái MULTIPLE và quản lý nút bật/tắt (Toggle Button) trên thanh player.
Yêu cầu Hàm:
renderChooserModal(candidates, onSelectCallback): Tạo một popup nhỏ lơ lửng góc màn hình hoặc giữa player. Khi người dùng click vào 1 item, nó sẽ gọi onSelectCallback(candidate).
destroyUI(): Xóa hoàn toàn popup rác ra khỏi DOM sau khi chọn xong hoặc khi video chuyển cảnh.
injectToggleButton(playerControls): Thêm một nút [ASS] cạnh nút CC của YouTube để người dùng chủ động tắt/mở overlay.

⚙️ 4. Chế độ Phân tách & Phân tích dữ liệu (parseMode Strategy)
Hệ thống linh hoạt vận hành dựa trên biến cấu hình `parseMode` trong Settings:
- Chế độ BACKGROUND (Mặc định):
  - Tầng nền (parser.js) chịu trách nhiệm bóc tách toàn bộ file `.ass` thô thành JSON Object sạch sẽ.
  - Chuyển đổi mã màu từ định dạng ASS (&HAABBGGRR) sang CSS RGBA lý tưởng.
  - Quy đổi toàn bộ thời gian dạng chuỗi (0:00:00.00) sang đơn vị mili-giây (ms) để tối ưu so khớp.
  - Tầng Content Script (renderer.js) nhận Object sạch, chạy bộ đếm thời gian cực nhẹ và render.
- Chế độ CONTENT:
  - Tầng nền chỉ tải văn bản và gửi chuỗi thô (rawText) về cho Content Script.
  - Content Script phải tự sở hữu hoặc gọi một bộ Parser nội bộ để tự xử lý chuỗi trên trình duyệt.


💾 5. Chính sách Bộ nhớ đệm (Cache Policy)
- Khóa định danh (Key): videoId (Chuỗi ID video của YouTube lấy trực tiếp từ URL).
- Giá trị lưu trữ (Value): Cấu trúc động tùy biến theo parseMode:
  {
    fileName: "sub_goc.ass",
    sourceType: "github" | "gdrive",
    cachedAt: 1715942400000,
    raw: "[Script Info]...", // Cấp phát trường này nếu parseMode là CONTENT
    parsedData: { info: {}, styles: {}, events: [] } // Cấp phát trường này nếu parseMode là BACKGROUND
  }
- Trình kích hoạt lưu cache (Cache Triggers):
  - Tự động lưu khi hệ thống thực hiện Auto-resolve (Chỉ tìm thấy 1 candidate duy nhất).
  - Lưu thủ công sau khi người dùng click chọn 1 phụ đề cụ thể từ danh sách hiển thị của UI Chooser.

💎 8. Quy tắc Fallback Style & Khai báo Style Default Nguyên Bản (Style Fallback Engine)
Quy tắc quản lý hiển thị khi gặp Style không xác định trong file phụ đề .ass:
- Khi một dòng Dialogue chứa một Style không tồn tại trong danh sách Styles của tệp phụ đề, Renderer.js phải thực hiện tra cứu và fallback về Style mang tên 'Default' được định nghĩa sẵn trong tệp đó.
- Trong trường hợp tệp phụ đề hoàn toàn không định nghĩa Style 'Default', Renderer.js sẽ tự động kích hoạt bộ lọc fallback cuối cùng bằng cách áp dụng cấu hình Object Style Default Nguyên Bản (được lưu trữ cố định ngay bên trong bộ nhớ của Renderer.js).
- Object Style Default Nguyên Bản tuân thủ chính xác các thông số từ cấu hình mẫu chuẩn hóa dưới đây:
